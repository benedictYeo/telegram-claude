**Constants:**

```typescript
const MAX_CONTEXT_MESSAGES = 40; // D-01: rolling window, 20 turns
```

**Function 1: `loadContext(chatId: number, resetHourUtc: number): ChatMessage[]`**
- Get DB via `getDb()`.
- Calculate daily reset timestamp via `getDailyResetTimestamp(resetHourUtc)`.
- Query: `SELECT id, created_at FROM conversations WHERE chat_id = ? AND archived = 0 ORDER BY id DESC LIMIT 1`.
- If conversation exists and `created_at < todayResetTs`: archive it silently (D-04) via `db.prepare("UPDATE conversations SET archived = 1 WHERE id = ?").run(conv.id)` and return `[]`.
- If no conversation: return `[]`.
- Query messages: `SELECT role, content FROM messages WHERE conversation_id = ? ORDER BY id ASC`.
- Apply rolling window trim: `rows.slice(-MAX_CONTEXT_MESSAGES)`.
- Return trimmed array as `ChatMessage[]`.

**Function 2: `appendMessages(chatId: number, userText: string, assistantText: string, inputTokens: number, outputTokens: number): void`**
- Get DB via `getDb()`.
- Wrap in `db.transaction(() => { ... })()` (Pitfall 2: race condition prevention).
- Inside transaction:
  1. Find active conversation: `SELECT id FROM conversations WHERE chat_id = ? AND archived = 0 ORDER BY id DESC LIMIT 1`.
  2. If none: create one: `INSERT INTO conversations (chat_id, created_at, archived) VALUES (?, ?, 0)` with `created_at = Math.floor(Date.now() / 1000)`. Get `lastInsertRowid`.
  3. Insert user message: `INSERT INTO messages (conversation_id, role, content, input_tokens, output_tokens, created_at) VALUES (?, 'user', ?, NULL, NULL, ?)`.
  4. Insert assistant message: `INSERT INTO messages (conversation_id, role, content, input_tokens, output_tokens, created_at) VALUES (?, 'assistant', ?, ?, ?, ?)`.
  5. Upsert daily usage: `INSERT INTO daily_usage (date, input_tokens, output_tokens) VALUES (?, ?, ?) ON CONFLICT(date) DO UPDATE SET input_tokens = input_tokens + excluded.input_tokens, output_tokens = output_tokens + excluded.output_tokens`. Date format: `new Date().toISOString().substring(0, 10)` (YYYY-MM-DD).

**Function 3: `archiveConversation(chatId: number): void`**
- Get DB via `getDb()`.
- `db.prepare("UPDATE conversations SET archived = 1 WHERE chat_id = ? AND archived = 0").run(chatId)`.
- Per D-05 and D-06: soft delete (archive), not hard delete.

**Function 4: `getUsage(chatId: number): { todayIn: number; todayOut: number; totalIn: number; totalOut: number; turns: number; lastTs: number | null }`**
- Get DB via `getDb()`.
- `const today = new Date().toISOString().substring(0, 10);`
- Query today's usage: `SELECT input_tokens, output_tokens FROM daily_usage WHERE date = ?` with `today`.
- Query total usage: `SELECT SUM(input_tokens) AS i, SUM(output_tokens) AS o FROM daily_usage`.
- Query conversation stats: `SELECT COUNT(*) AS turns, MAX(m.created_at) AS last_ts FROM messages m JOIN conversations c ON m.conversation_id = c.id WHERE c.chat_id = ? AND c.archived = 0`.
- Return object with all fields, defaulting to 0/null for missing data.

**Private helper: `getDailyResetTimestamp(resetHourUtc: number): number`**
- Per D-02: daily reset at specified UTC hour (default 21 = 5am SGT).
- Create a Date at today's reset hour UTC.
- If that timestamp is in the future, subtract one day.
- Return as Unix timestamp (seconds).

NOTE: `buildSystemPrompt()` is NOT in this module. Per the Architectural Responsibility Map, system prompt assembly belongs in the Core layer (`src/core/router.ts`). It will be created in Task 2.

Follow project conventions: import type for Env if needed (but this module may not need Env -- it uses getDb() directly). Use `.js` extension on imports. Double quotes. Named exports only.

**Export `chat()` function:**
```typescript
export async function chat(
  messages: ChatMessage[],
  systemPrompt: string,
  env: Env,
): Promise<ChatResult>
```

Implementation:
1. Validate `env.ANTHROPIC_API_KEY` is non-empty. If empty, `throw new Error("ANTHROPIC_API_KEY is not set")` (Pitfall 3 from RESEARCH.md -- fail early with descriptive error).
2. Create client: `const client = new Anthropic({ apiKey: env.ANTHROPIC_API_KEY })`.
   - Per-call instantiation, not module singleton (API key flows through env at runtime).
3. Determine model: `const model = env.DEFAULT_MODEL || "claude-haiku-4-5-20251001"` (D-15, D-16).
4. Call `client.messages.create()` with:
   - `model`: from above
   - `max_tokens: 4096`
   - `system: systemPrompt` (top-level parameter, NOT a message role -- Pitfall 4)
   - `messages`: pass through directly
   - Comment for extended thinking: `// D-17: Extended thinking infrastructure -- enable when thinking-capable models are added in Phase 3`
   - Do NOT enable extended thinking (D-17 says wire infrastructure, disabled by default; Haiku 4.5 does not support it well).
5. Extract text from response: `response.content.filter(b => b.type === "text").map(b => b.text).join("")`.
6. Return `{ text, inputTokens: response.usage.input_tokens, outputTokens: response.usage.output_tokens }`.

Do NOT catch errors in this function. Let errors bubble to the handler's catch block (matches existing core layer pattern from telegram.ts -- no try/catch in core functions).

Per D-18: non-streaming API call. Single call, full response.

Follow project conventions: double quotes, 2-space indent, semicolons, `.js` extension on local imports. No default export.
